# 化妆品工厂 GEO 优化系统

让"关于这家工厂的问题"在豆包 / 通义千问 / DeepSeek / 文心 / 元宝 / Kimi 的回答里
被**提及 → 推荐 → 给链接**。服务对象：国内工厂文案手。

> 本系统参照 CSDN 文章 (ximenjianxue/159200646) 的三模块框架搭建，但做了一个关键改造：
> 把文章里"用户点击/停留越多权重越高"的反馈信号，**换成"AI 引用越多则该方向越该加码/缺口越大越该优先"**，
> 使它从"站内搜索排序"真正变成"生成式引擎优化"。详见下方映射表。

## 两层架构

```
┌─────────────────────────────────────────────────────────┐
│  工具层 tool/  （技术方跑，复刻文章三模块）                │
│                                                           │
│  queries.csv ──┐                                          │
│                ├──► geo_engine.py ──► topic_ranking.csv   │
│  citations.csv ┘     (语义聚类+引用缺口排序)   (选题优先级) │
│                                                           │
│  hot_seeds.csv ──► hot_keywords.py ──► 新问题入库          │
└───────────────────────────┬─────────────────────────────┘
                            │ 产出"下一篇该写什么"
                            ▼
┌─────────────────────────────────────────────────────────┐
│  工作流层 workflow/  （文案手照着做）                      │
│                                                           │
│  01问题库 → 02写作SOP → 04评分卡 → 03分发对照              │
│       ▲                                    │              │
│       │         06月度复盘 ◄── 05监测看板 ◄┘              │
│       └────────── 引用数据回流闭环 ──────────              │
└─────────────────────────────────────────────────────────┘
```

工具层产出"选题优先级"喂给文案手；文案手发布后把引用数据填回监测看板，
回流成 `citations.csv`，下次跑引擎时优先级自动更新。这就是闭环。

## 文案手日常 7 步

| 步 | 做什么 | 用哪个文件 |
|---|---|---|
| 1 | 看今天写什么 | `tool/data/topic_ranking.csv` 最上面的选题 |
| 2 | 没数据时先跑基线 | 拿问题去各引擎问一遍，记下被不被提及 |
| 3 | 按规范写 | `workflow/02-写作SOP.md` 六条铁律 |
| 4 | 发布前自评 | `workflow/04-评分卡.md`，≥34分才发 |
| 5 | 一稿多发 | `workflow/03-引擎检索源对照.md` 按平台改写 |
| 6 | 回收引用 | 隔1–4周重跑问题，填 `workflow/05-监测看板.csv` |
| 7 | 月底复盘 | `workflow/06-月度复盘模板.md` |

## 技术方怎么跑工具层

```bash
cd tool

# 1. 算选题优先级（核心）。读 queries.csv + citations.csv，出 topic_ranking.csv
python geo_engine.py

# 2. 把行业热词派生成新问题入库（可选）。先预览：
python hot_keywords.py --dry-run
python hot_keywords.py          # 确认无误后真正写入

# 依赖：jieba, scikit-learn, numpy（已装）。全程无需联网、无需 API key。
```

**冷启动**：第一次没有引用数据时，引擎自动进入冷启动模式，
优先级 = 纯业务价值，先按它去各引擎跑基线，回填 `citations.csv` 后再跑一次即可。

## 与原文章的模块映射

| 文章原模块 | 原本干嘛 | 本系统改造为 |
|---|---|---|
| ① 关键词语义分析 | 关键词向量化+余弦相似度 | jieba+TF-IDF 把问题库**语义聚类**，一篇覆盖一簇 |
| ② 检索权重动态调整 | 点击/停留越多权重越高 | **AI 引用反馈**算覆盖度，缺口越大越优先 |
| ③ 用户行为反馈闭环 | 行为数据回流调参 | 监测看板→月度复盘的引用闭环 |
| ④ 热点关键词抓取 | 接百度指数/微博热搜 | 本地热词种子派生问题（真实API需自备key） |
| 输出：GEO排序 | 搜索结果排序 | 选题优先级排序 topic_ranking.csv |

**没照搬的部分**：文章的 SQLite 向量存储、Spring Boot+Vue 后台、GeoTools 地理逻辑——
那些是"站内搜索引擎"的技术栈，与文案手做 GEO 无关，已舍弃。核心公式
`score = similarity × weight` 被保留并改造为 `优先级 = 业务价值 × 引用缺口`。

## 🔧 换行业（非化妆品）只改三处

1. **问题库**：`tool/data/queries.csv` 的 query 列换成你行业用户会问的话
2. **权威信号**：`workflow/02-写作SOP.md` 和 `04-评分卡.md` 的"权威信号"换成你行业的资质/认证/检测
3. **目标平台**：`workflow/03-引擎检索源对照.md` 按你客户常用引擎重选 2–3 个主战场

改完重跑 `python tool/geo_engine.py` 即可。框架完全通用。

## 重要提醒

- **检索源数据是易变信息**：`03-引擎检索源对照.md` 基于 2026 初单行业样本，
  落地第一周务必用本厂真实问题复测，每月更新。
- **API 与 App 行为可能不一致**：未来若自动化监测，API 版联网检索结果可能和
  手机 App 不同，需保留人工抽查校准。
- **先轻后重**：先用飞书/Notion 多维表 + 人工监测跑通方法论（1个月），
  验证有效再上自动化脚本。
